# Output Format

Template chính thức cho báo cáo `/vbs-scan-security`. SKILL.md và workflows phải tuân thủ format này. Mọi user-facing text dùng key i18n (xem [`i18n/vi.md`](i18n/vi.md) và [`i18n/en.md`](i18n/en.md)) — KHÔNG hardcode.

## Cấu trúc tổng quan (v0.3+)

```
1. Header block (metadata)
2. Verdict line (PASS/WARN/FAIL + 1-line description)
3. Section: CRITICAL — Overview (bảng compact) → Verbose blocks per finding
4. Section: HIGH — Overview (bảng compact) → Verbose blocks per finding
5. Section: MEDIUM (bảng compact, 1 dòng per finding)
6. Section: LOW (bảng compact, 1 dòng per finding)
7. Section: PASSED CHECKS (checklist)
8. Section: Next steps (1-2 dòng)
9. Footer (generated by, disclaimer, save location)
10. JSON summary (canonical EN, fenced ```json) — cho machine consumption
```

**Verbose level theo severity (v0.3+):**

| Severity | Format |
|---|---|
| CRITICAL | Bảng overview + full block (Mô tả ngắn + Tại sao nguy hiểm + Hacker khai thác + Code before/after + Đọc thêm) |
| HIGH | Bảng overview + medium block (Mô tả + Tác động + Code fix + Đọc thêm) |
| MEDIUM | Chỉ bảng compact (File:Dòng, Loại, Mô tả) |
| LOW | Chỉ bảng compact (File:Dòng, Loại, Mô tả) |

Mục đích: non-tech user đọc CRITICAL phải hiểu được tác hại + cách sửa mà không cần search Google.

### 🔴 [1] CRITICAL — SQL-INJECTION tại `api/users.ts:42`

**{header_short_description}:** Code ghép `req.body.id` từ user vào câu SQL không qua bảo vệ.

**{header_why_dangerous}**

Đây là lỗ hổng số 1 trong lịch sử web. Hacker chỉ cần gõ vài ký tự vào URL/form là có thể: dump toàn bộ user/password, xóa cả database, hoặc cài backdoor để chiếm server. OWASP xếp SQL injection là A03 trong Top 10 (2021). Trong 5 năm qua, các vụ rò rỉ dữ liệu lớn nhất (Equifax, LinkedIn, Yahoo) đều có liên quan đến SQL injection.

**{header_attack_scenario}**

1. Mở browser truy cập: `https://yoursite.com/users?id=1' OR '1'='1`
2. Code chạy thành câu SQL: `SELECT * FROM users WHERE id = 1' OR '1'='1`
3. Điều kiện `'1'='1'` LUÔN ĐÚNG → DB trả về **TẤT CẢ users** (kể cả admin)
4. Tiếp tục `id=1 UNION SELECT password,email FROM users` → dump toàn bộ password
5. Thời gian: 5-15 phút để có toàn bộ DB

**{header_current_code}**

\```javascript
const id = req.body.id  // {phrase_input_untrusted}
db.query(`SELECT * FROM users WHERE id = ${id}`)  // {phrase_concat_to_sink} SQL
\```

**{header_safe_code}**

\```javascript
const id = req.body.id
db.query('SELECT * FROM users WHERE id = ?', [id])  // dấu ? là placeholder, DB tự xử lý input an toàn
// hoặc với ORM (an toàn mặc định):
const user = await User.findOne({ where: { id } })
\```

**{header_read_more}:** [Rule chi tiết](skills/vbs-scan-security/rules/generic/02-sql-injection.md) · [OWASP A03 Injection](https://owasp.org/Top10/A03_2021-Injection/)

### 🔴 [2] CRITICAL — HARDCODED-SECRET tại `.env:1`

**{header_short_description}:** Stripe live key đã commit vào repo, ai clone hoặc xem source là thấy.

**{header_why_dangerous}**

GitHub có bot quét 24/7 tìm secret lộ. Khi bạn `git push` file `.env`, trong vài giây-vài phút bot tìm thấy key này. Với Stripe live key, kẻ tấn công có thể tạo charge giả, drain tài khoản; với AWS key, chiếm hết infrastructure và charge max billing.

**{header_attack_scenario}**

1. Repo push lên GitHub public
2. Bot (truffleHog, gitleaks, GitGuardian, hoặc bot riêng của attacker) scan commit mới trong vài giây
3. Match regex `sk_live_...` → extract key
4. Test key qua Stripe API → confirm live
5. Tạo charge to attacker's account, hoặc tạo refund tunneling
6. Thời gian từ push → exploit: **dưới 5 phút**

**{header_current_code}**

\```bash
# .env (committed to git)
STRIPE_KEY=sk_live_EXAMPLE_REPLACE_WITH_REAL_KEY_IN_PROD
\```

**{header_safe_code}**

\```bash
# 1. XOAY KEY NGAY (assume nó đã bị compromise)
# 2. Xoá .env khỏi git history:
git rm --cached .env
echo ".env" >> .gitignore
git add .gitignore
git commit -m "Untrack .env"
# 3. Force rewrite history nếu repo còn private hoặc clone mới:
# git filter-repo --path .env --invert-paths
# 4. Set key qua env var hoặc secrets manager (AWS Secrets Manager, Vault, Doppler)
\```

### 🟠 [4] HIGH — WEAK-PASSWORD-HASHING tại `auth.ts:18`

**{header_short_description}:** Mật khẩu hash bằng MD5 — bị bẻ khóa trong vài giây bằng GPU/rainbow table.

**{header_impact}**

Nếu database bị lộ (qua SQL injection, backup leak, hoặc nội bộ làm tệ), TOÀN BỘ password user lộ trong vài giờ. User dùng cùng password ở các site khác cũng bị compromise (credential stuffing — đây là cách 80% account takeover xảy ra trong thực tế).

**{header_fix_code}**

\```javascript
// THAY:
const hash = crypto.createHash('md5').update(password).digest('hex')

// BẰNG bcrypt (chuẩn industry):
const bcrypt = require('bcrypt')
const hash = await bcrypt.hash(password, 12)  // cost factor 12 = chậm vừa đủ để hacker không brute-force được

// HOẶC argon2 (modern, OWASP recommend):
const argon2 = require('argon2')
const hash = await argon2.hash(password)
\```

## Quy tắc render

### 1. Section nào không có finding thì bỏ section đó

Ví dụ nếu không có CRITICAL: bỏ luôn heading `## {section_critical}` và bảng tương ứng. Đừng render bảng rỗng.

### 2. Severity cap theo rule

Mỗi rule trong `rules/generic/*.md` có `severity_max` ở frontmatter. Khi flag, severity gán cho finding không được vượt `severity_max`. Có thể giảm xuống nếu context cụ thể ít nguy hiểm hơn.

Ví dụ: rule `XSS` có `severity_max: HIGH`. Nếu phát hiện XSS chỉ ở admin-only page có auth thì có thể downgrade xuống MEDIUM. Phải ghi rõ lý do downgrade vào cột Issue.

### 3. File:Line format

- Cố gắng cho ra số dòng chính xác. Nếu rule áp dụng cho cả file (vd `.env` committed): dùng `:1`.
- Nếu finding spread nhiều dòng: dùng `:start-end` (vd `routes.ts:42-58`).
- Path tương đối với git root, dùng `/` (cả trên Windows).

### 4. Issue column ngắn gọn

Tối đa 1 dòng, có thể include 1 code snippet ngắn (~60 ký tự) trong backtick. Code snippet KHÔNG bị dịch sang lang khác.

### 5. Fix column

Ưu tiên dùng phrase template từ i18n (`{phrase_use_parameterized}`, `{phrase_use_bcrypt}`...). Nếu cần custom message, viết theo lang chọn nhưng để tên technical (function name, library name) NGUYÊN tiếng Anh.

### 6. PASSED section

Liệt kê các rule mà sau khi quét **không phát hiện vấn đề**. Format:
```
- ✓ <Rule Name> — <ngắn 1 dòng giải thích đã check gì>
```

Không cần cho rule không applicable (vd: PHP rules trên repo Node — không cần liệt kê là "đã pass").

### 7. JSON summary cuối

**LUÔN dùng key tiếng Anh canonical**, bất kể `lang=vi` hay `en`. Đây là phần để CI/tooling parse. Các field:

| Field | Type | Description |
|---|---|---|
| `verdict` | `"PASS"\|"WARN"\|"FAIL"` | Tổng kết |
| `summary.critical` | int | Số **findings** (không phải rule-hit) ở mức CRITICAL |
| `summary.high` | int | Số findings ở mức HIGH |
| `summary.medium` | int | Số findings ở mức MEDIUM |
| `summary.low` | int | Số findings ở mức LOW |
| `summary.passed` | int | Số rule đã check và pass (max 21 trong rules generic) |
| `scope` | string | Scope label (canonical EN) |
| `files_reviewed` | int | |
| `primary_language` | string | lowercase, e.g. `"typescript"`, `"go"`, `"php"` |
| `specialized_rules_used` | bool | |
| `mode` | `"small"\|"large"` | |
| `date` | ISO 8601 date | |
| `findings[]` | array | Mỗi finding: file, line, rule_id, severity, issue_summary, fix_summary |
| `top_rules_by_count[]` | array (optional) | Optional ranking; mỗi entry: `{rule_id, count}` |

**Quy tắc đếm findings — QUAN TRỌNG để tránh sai số:**

1. **Mỗi finding có ĐÚNG 1 `rule_id`** (primary). Không gom nhiều rule_id vào 1 finding.
2. **1 dòng code có thể tạo nhiều findings nếu trigger nhiều rule.** Ví dụ `routes/wallet.ts:12` có cả IDOR và RACE-CONDITION → tạo 2 findings riêng biệt, mỗi cái có rule_id của mình.
3. **`summary.critical/high/medium/low` = số ENTRY trong `findings[]`** lọc theo severity. Tổng `critical + high + medium + low = findings.length`.
4. **`top_rules_by_count[]` (nếu render) = group `findings[]` theo `rule_id`, count.** Tổng counts = `findings.length` (không double-count).
5. Cross-references trong `issue` cho finding khác (e.g., "see also XSS at line X") là **text note**, KHÔNG tạo finding thứ 2.

**Sanity check khi render:**
```
assert findings.length == summary.critical + summary.high + summary.medium + summary.low
assert findings.length == sum(top_rules_by_count[].count)  # nếu có render top_rules
```

Nếu sanity check fail → có dedup/count bug ở aggregate step. Re-check trước khi output.

### 8. Footer disclaimer (bắt buộc)

Mọi report đều có disclaimer `{footer_disclaimer}` — vbsec là tool tham khảo, KHÔNG thay thế audit chuyên nghiệp. Người dùng có thể tin tưởng vbsec để catch lỗi phổ biến nhưng không nên dùng làm proof bảo mật pháp lý.

## ĐÃ ĐẠT

- ✓ XSS — Vue auto-escape, không phát hiện innerHTML với user input
- ✓ CORS — Origin cụ thể, không dùng `*`
- ✓ CSRF — Token CSRF có trong form
- ✓ IDOR — Route `/api/orders/:id` có authz check
- ✓ Insecure Deserialization — Không dùng pickle/unserialize/eval
- ✓ Path Traversal — Không có file ops với user input
- ✓ SSRF — Server không fetch URL user-controlled
- ✓ JWT None — Không thấy JWT, không applicable
- ✓ Unrestricted File Upload — Không có upload endpoint trong scope
- ✓ Race Condition — Không thấy balance update / inventory logic
- ✓ Slopsquatting — Tất cả import packages có trong npm registry
- ✓ Mass Assignment — Endpoint dùng explicit field selection
- ✓ Command Injection — Không có exec/spawn với user input
- ✓ Verbose Error / Debug Mode — Production config tắt debug

## Edge cases

| Scenario | Cách render |
|---|---|
| Repo PASS hoàn toàn (no findings) | Hiện header + verdict PASS + section PASSED + footer. Không có section CRITICAL/HIGH/MEDIUM/LOW. |
| Scope rỗng (no files) | Render header với "Files: 0" + msg "{msg_no_files}". Bỏ qua tất cả section findings. Verdict = PASS (mặc định). |
| Multi-lang repo | `header_primary_lang` ghi 2 lang, e.g. `"go + vue (cả 2 đều dùng specialized)"`. |
| Rule không applicable | Không đưa vào PASSED list, không hiện. Ví dụ: PHP rules trong repo thuần Node. |
| Sub-agent (LARGE mode) finding bị conflict | Main agent dedup theo `file:line:rule_id`. Giữ severity cao nhất. |
